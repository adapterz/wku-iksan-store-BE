# API 설계 리뷰 회의 정리

## 1. API 응답(Response) 템플릿

- 응답 형태(response)는 통일된 정갈한 포맷으로 맞출 것
- status 코드를 세분화하여 분기 로직의 기준으로 삼는 방식은 지양
  - (예시) 200을 리턴하더라도 FE에서 body를 fetch 후 파싱하여, 그 안의 상태값을 기준으로 화면을 분기 처리할 수 있음
  - 즉 HTTP status code 자체보다 응답 body 안의 구조(스키마)를 통일하는 것이 중요
- 상세 조회 응답은 배열(`[ ]`)이 아닌 단일 객체로 반환
- 응답 포맷은 하나로 정하고 전체 API에 일관되게 적용

## 2. 도메인/네이밍 정리

- 회원가입 → **auth(인증)** 도메인으로 변경
- 회원 → **인증**으로 명칭 통일
- **유저 도메인** 추가 필요
- 주문서 → **주문**으로 명칭 통일

## 3. 상품 목록 조회 (API #5)

- 배열 하나가 누락되어 있음 → 확인 필요
- 가상 브랜드까지 응답에 포함
- `product` 테이블 신규 생성
  - `product` 테이블에 `brand_name` 컬럼 추가
  - 좋아요 개수 필드 추가 (액션 기능이 없더라도 카운트는 응답에 포함)

## 4. 상품 상세 조회 (API #6)

- 배송비 포함/미포함 여부 반환
- 배송비 금액 반환
- 후기 개수 반환
- 이미지 다중 등록 대응 필요 (현재 image 필드가 1개라 여러 장 등록 시 분리 불가 → 구조 개선 필요)

> ⚠️ 모든 기능은 CRUD로 구성되어야 하는데, #5·#6 항목에서 CRUD 중 일부가 누락되어 있음. 확인 및 보완 필요.

## 5. 주문 생성 (API #7)

- CREATE 성격의 API로 정의
- 구현 시 상세 로직 추가 고민 필요

## 6. 메시지 관련 (API #9)

- message 필드에 한글 사용 금지
- **주문 상태 전이(state transition) 설계 필요**
  - 흐름 예시: 주문 전(주문 없음) → 주문 완료 → `pending` 상태 → 이후 어떤 상태로 전이되는지 불명확
  - 예: 카카오 가입 신청처럼 매니저 승인이 필요한 구조에서, 상태값이 3개뿐이라면 각 상태 간 전이 가능 경로를 명확히 정의해야 함
  - **상태값 목록과 상태 전이 규칙을 명시적으로 정의할 것**

## 7. 테이블/리소스 구조

- 테이블: `user`, `order`, `gift`, `product`
- 리소스: 상품(product), 유저(user)
  - 식별자: `user_id`, `product_id`, `order_id`, `gift_id`
- `user_id`가 `product_id`를 참조하여 유저가 보유한 상품을 조회
- `gift` 테이블 구성: `product_id`, `sender_id`, `status_code`(예: used) 등 추가

## 8. 결제 완료 처리 (API #10)

- 결제 완료 시 **삭제(DELETE)** 처리

## 9. 바코드 기능 (API #13)

- 바코드 자체 구현은 불필요
- 이미지 형태로 반환만 되도록 구현

## 추가 힌트 — 설계 프로세스

> 프로젝트 개요에 전체 내용이 명시되어 있음. 주문 발생 시 **받는 사람이 누구인지는 중요하지 않음** (ID 값만 바꾸면 본인으로도 변경 가능하므로).

### 설계 진행 순서 (M0 단계 핵심)

1. Figma에서 필요한 페이지 선정
   - 홈 화면
   - 상품 설명 (나에게 선물하기 / 선물하기 기능만 구현)
   - 로그인
   - 수량 선택
   - 선물 상자
   - 주문하기 전환 (메시지 작성 기능)
   - 메시지 편집
   - 주문 완료
   - 선물함
   - 선물함 사용하기
2. 각 페이지별로 필요한 API 구분
3. API에 맞춰 ERD 설계 요청

> 1~3번이 현재 M0 단계에서 가장 중요한 작업

### 유의사항

- 주니어가 가장 많이 하는 실수: 바꿀 부분이 많다고 판단되면 처음부터 새로 갈아엎는 것 → 지양
- 설계를 팀원 각자 따로 진행하면 기획 방향이 어긋날 수 있음
- 기획은 팀의 공통 목표이므로 **비동기 커뮤니케이션의 장점(기록이 남는다)**을 최대한 활용할 것
- 답변을 기다리며 작업을 멈추는 방식은 비동기 소통을 동기 소통처럼 사용하는 것 → 비동기 소통의 장점을 살리는 방향으로 진행
