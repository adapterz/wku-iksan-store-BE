## 1. 배포를 맡았을 때 실제로 어떤 일을 했는가

배포 담당으로서 한 일을 시간 순서대로 정리하면 아래와 같습니다.

### 1) 레포 기본 세팅부터 시작

가장 먼저 GitHub 레포 상태를 점검했습니다. `.gitignore`가 비어있고 `.env`가 이미 git에 커밋돼 있는 걸 발견해서, `.gitignore`(node_modules, .env 등)를 채우고 `.env.example`(실제 값 없이 항목 이름만) 만드는 것부터 시작했습니다. 이 작업 하나도 `feature/deploy-env` 브랜치를 새로 만들어서 진행 → PR → 팀원 승인 → `develop` 머지 → 브랜치 삭제, 이 사이클을 그대로 따랐습니다.

### 2) AWS 인프라 준비

- EC2 인스턴스 1대 생성 (팀 전체가 공유하는 단일 서버 원칙)
- 보안그룹에서 SSH(22), 웹(3000), MySQL(3306) 포트 개방
- EC2 안에 MySQL 8 설치 → 팀 공용 DB(`iksanshop`)와 계정(`teamuser`) 생성, `iksanshop.*`에 권한 부여
- `bind-address`를 `0.0.0.0`으로 바꿔서 외부(팀원 로컬)에서도 DB 접속 가능하게 설정

### 3) 서버 실행 환경 구성

- EC2에 SSH로 접속해서 Node.js 24 LTS(NodeSource 저장소 이용), pm2를 설치
- `ecosystem.config.js` 작성 (이것도 배포 담당 몫으로, `app.js`를 pm2가 관리하도록 하는 설정) → 별도 브랜치(`feature/deploy-pm2`)로 작업해서 PR·머지

### 4) 실제 배포 실행

- EC2에 레포를 `git clone`하고 `develop` 브랜치로 전환
- EC2 안에서 `.env` 파일을 직접 생성 (git엔 없는 파일이라 서버에서 새로 만들어야 함), `SESSION_SECRET`은 랜덤 문자열로 생성해서 채움
- `npm install` → `pm2 start ecosystem.config.js` → `pm2 status`로 정상 기동 확인

### 5) 배포 확인 중 발견한 문제 대응

- 브라우저로 공인 IP 접속했을 때 "상품목록을 불러오는데 실패했습니다" 에러 확인
- `pm2 logs`로 실제 에러(`no such table`)를 확인해서, 서버 코드 문제가 아니라 **DB 테이블 자체가 없다**는 걸 원인으로 좁힘 

### 6) 안정성 작업

- `pm2 save` + `pm2 startup`으로 EC2 재부팅 시에도 서버가 자동으로 살아나도록 설정
- 퍼블릭 IP가 "중지 후 시작" 시 바뀔 수 있다는 점을 파악하고 Elastic IP(고정 IP) 필요성 검토

### 7) 문서화로 마무리

- `README.md`에 "EC2 접속 → 코드 배치 → `.env` 준비 → pm2 실행 → 배포 확인"까지 전체 절차를 팀원 누구나 따라할 수 있게 기록
- `docs/Cloud/DEVLOG.md`에 날짜별 작업 내용을 한 줄씩 기록
- 이 문서(인수인계 기록)를 작성

## 2. 전체 작업 순서 요약 (한눈에 보기)

1. GitHub 레포 정리 — `.gitignore`, `.env.example` 작성
2. AWS EC2 인스턴스 생성, 보안그룹 설정 (22·3000·3306 등)
3. EC2 안에 MySQL 8 설치 → 팀 공용 DB, 계정 생성
4. EC2 안에 Node.js 24 LTS, pm2 설치
5. `ecosystem.config.js` 작성 (pm2 실행 설정)
6. EC2에 레포 clone → `.env` 생성 → `npm install` → `pm2 start`
7. 브라우저로 공인 IP 접속 확인
8. `README.md`에 재배포 절차 문서화

## 3. 겪었던 문제와 해결


### 문제 1 — Node.js 설치 중 SSH 연결이 끊김

- **증상**: EC2에서 `apt-get update` / Node 설치 스크립트 실행 중 `client_loop: send disconnect: Connection reset by peer` 발생
- **원인**: 프리티어(t2.micro 등) 인스턴스는 RAM이 1GB뿐이라, 패키지 설치 중 메모리 압박으로 OOM(메모리 부족) 킬러가 SSH 세션을 끊음
- **해결**: 재접속 후 재시도
- **배운 점**: 프리티어 인스턴스에서 패키지 설치할 땐 연결이 끊길 수 있다는 걸 미리 감안하고, 끊기면 바로 재접속해서 이어가면 됨 — 당황할 필요 없음

### 문제 2 — 배포 후 "상품목록을 불러오는데 실패했습니다"

- **증상**: pm2로 서버는 정상 기동됐는데, 브라우저에서 상품 목록이 안 뜸
- **원인**: `pm2 logs`로 확인해보니 `no such table` 에러 — `db/schema.sql`, `db/seed.sql` 파일이 여전히 빈 파일(0바이트)이라 실제 DB에 테이블 자체가 없었음
- **해결**: DB 담당에게 스키마 작업 진행 상황 확인 요청. 서버 코드(BE)는 이미 완성됐어도, DB 스키마가 준비 안 되면 배포해도 동작 안 함을 확인
- **배운 점**: "배포가 됐다"와 "기능이 동작한다"는 다른 문제. `pm2 status`가 `online`이어도 실제 API 호출까지 확인(`pm2 logs`)해야 진짜 문제를 찾을 수 있음. 역할 간 의존성(BE는 DB 스키마가 있어야 동작)을 미리 인지하고 있어야 함

### 문제 3 — DB 담당이 EC2 MySQL에 직접 접속해 라이브로 테이블 작성 중

- **증상**: 실제 DB에는 테이블이 생기고 있는데, git의 `db/schema.sql`은 여전히 비어있는 불일치 상태
- **원인**: DB 작업을 git 파일에 먼저 작성 → 적용하는 순서가 아니라, MySQL 클라이언트로 라이브 DB에 바로 작업
- **해결(진행중)**: `mysqldump --no-data`로 완성된 스키마를 다시 파일로 뽑아서 git에 커밋하도록 안내
- **배운 점**: 공용 DB 하나를 여러 명이 만지는 구조이다 보니, "실제 DB 상태"와 "git에 적힌 공식 스키마"가 어긋나기 쉬움. 라이브로 작업했더라도 반드시 파일로 다시 남겨야 팀 전체가 같은 기준을 갖게 됨

### 문제 4 — 3306 포트를 `0.0.0.0`으로 연 것이 위험해 보였음

- **증상**: 처음엔 MySQL 포트를 전 세계에 열어둔 게 보안 문제라고 판단해 닫으려 함
- **원인**: 사실은 팀 설계상 **의도된 것**이었음 — DB 담당이 본인 로컬 컴퓨터에서 EC2의 MySQL로 직접 원격 접속해 스키마를 적용해야 하는 구조라, 외부 접속이 필요했음
- **해결**: 팀 가이드 문서를 재확인하고 오해를 정정. 다만 `.pem` 키는 배포 담당만 보관, DB 접속정보는 내부 채널로만 공유하는 원칙은 유지
- **배운 점**: 보안 원칙(포트를 최소한만 열기)이 항상 정답은 아니고, **팀의 실제 협업 구조**(누가 어디서 무엇에 접속해야 하는지)를 먼저 파악한 뒤 판단해야 함. 지레짐작으로 "닫아야 한다"고 단정하지 말 것

## 4. 앞으로 할 사람이 알아두면 좋은 것

- **EC2 재부팅/중지 시**: `pm2 startup` + `pm2 save`를 미리 해두면 재부팅돼도 서버가 자동으로 살아남음. 안 해두면 매번 SSH 접속해서 수동으로 `pm2 start` 해줘야 함
- **퍼블릭 IP는 "중지 후 시작" 시 바뀔 수 있음** (재부팅은 유지됨) — Elastic IP(고정 IP)로 바꿔두면 이 문제 자체가 사라짐
- **SQL 실행은 EC2 접속 없이도 가능** — 3306이 열려있어서, 로컬 컴퓨터에서 `-h <EC2 IP>`로 원격 실행하면 됨. EC2는 코드(BE 서버) 배치할 때만 접속 필요
- **커밋 컨벤션**: `종류: 제목` (`chore:`, `docs:`, `fix:` 등) — 안 지키면 나중에 리뷰 때 다시 고쳐야 함(경험담)

---

## 5. 느낀 점

- 배포는 "설정 한 번 하고 끝"이 아니라, 다른 역할(BE·DB)의 진행 상황에 따라 계속 재확인해야 하는 작업이라는 걸 체감함
- 에러 메시지(`no such table`, `Connection reset by peer` 등)를 정확히 읽고 원인을 좁혀가는 게 결국 가장 빠른 해결법이었음 — 막연히 "재설치"부터 하지 않아서 시간을 아낌
- 팀 문서(가이드)를 먼저 확인하고 판단하는 습관이 중요함
