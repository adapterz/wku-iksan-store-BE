## 주문하기 (P_order) 페이지

## API 설계구현 필요기능

| 기능 | 필요 API | 이유 |
| --- | --- | --- |
| 메시지 입력(편집) | `POST /api/orders` 요청 바디의 `message` 필드 | 사용자가 직접 입력한 값이 실제로 주문 데이터로 DB에 저장되어야 하는 실질 기능. 화면 표시가 아니라 서버에 남는 데이터이므로 API 연동 필수 |
| "결제하기" 버튼 | `POST /api/orders` | 주문을 최종 확정·전송하는 실제 트리거. 이 호출로 `orders`·`gifts` 레코드가 생성되어야 M2 핵심 흐름(주문 생성→선물 발급)이 완성됨 |
## API구현 불필요기능

| 기능 | 이유 |
| --- | --- |
| "선물하기" 탭 | M2는 선물 모드 하나뿐 — 탭 전환 로직 자체가 불필요 |
| 기본 메시지 카드 디자인 | 정적 UI 장식, 실제 데이터 처리 없음 |
| 상품 이미지·브랜드명·상품명·상품금액 표시 | 상품설명 페이지에서 이미 `GET /api/products/:id`로 조회한 데이터를 그대로 넘겨받아 표시만 하는 것 — 이 페이지에서 재조회할 필요 없음 |
| 배송지 표시 / 변경하기 / 배송 요청사항 | M2 상품은 전부 교환권(QR)형이라 실물배송·배송지 개념 자체가 없음 |
| 쇼핑포인트 사용 / 입력 / 전액사용 / 충전 이동 | 포인트 시스템은 M3 선택 기능이며 `users` 테이블에 포인트 컬럼 자체가 없음 |
| 카카오톡 잔액 표시·잔액부족 검증 | 결제는 MOCK — 검증할 실제 잔액 데이터가 없는, 하드코딩된 보여주기용 값 |
| 할인금액 | 쿠폰/할인 시스템은 M2 범위 밖(부가 기능) |
| 최종 결제금액 계산식(할인·포인트 반영) | 할인·포인트가 없으므로 상품금액과 항상 동일 — 별도 계산 로직 불필요 |
| 이용약관 체크박스 | 법적 UI 장식일 뿐 주문 생성 로직에 영향 없음 |

## 결론

- 상품 정보(이미지·이름·가격)는 상품설명 페이지에서 이미 조회·전달받은 데이터를 재사용하므로 이 페이지에서 별도 조회 API가 필요 없습니다.
- 배송지·포인트·할인·잔액검증·약관동의는 M2 DB 스키마(`users`·`products`·`orders`·`gifts`)에 대응하는 테이블/컬럼이 없어 애초에 구현이 불가능하거나, MOCK 결제 특성상 검증할 실체가 없는 보여주기용 UI입니다.
- 따라서 주문하기 페이지는 "메시지 입력 폼 + 결제하기 버튼" 정도의 최소 구성으로 구현하고, `POST /api/orders`가 정상 동작해 `orders`·`gifts` 레코드를 생성하는 것에 개발 리소스를 집중하는 것이 M2 목표(끊김 없는 전체 흐름 시연)에 부합합니다.
