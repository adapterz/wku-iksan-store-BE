## 10. M2 후반 트러블슈팅 및 PR 검토 체계 확립

> M2 1~6단계 구현과 배포 안정화를 마친 이후에도, FE와 다른 직무의 PR이 이어지면서 BE 영역과 겹치거나 이전에 해결했던 문제가 다른 브랜치에서 재발하는 사례가 반복되었다. 이 시기부터는 "새로운 기능을 만드는 작업"보다 "이미 만든 기능을 지키고, PR을 체계적으로 검토하는 작업"의 비중이 커졌다.

### 작업 내용

- `.env` 유실, 세션 인증 옵션 누락, 가격 조작 취약점 등 반복되는 문제 대응
- 서로 다른 브랜치에서 동일한 버그가 재발하는 것을 방지
- 유실된 API 기능을 git 이력에서 복구
- PR 검토 시 활용할 자체 방법론 확립

### 진행 과정

1. develop → main merge 과정에서 `.env` 파일이 삭제되는 커밋이 반영되어 팀 전체와 개인 로컬 환경에서 반복적으로 서버 500 에러가 발생, 매번 `.env` 재생성으로 대응
2. FE PR(로그인/회원가입, 주문 페이지)에서 세션 쿠키 전송에 필요한 `credentials: 'include'` 옵션이 반복적으로 누락되는 것을 발견하고 수정 요청
3. FE가 `POST /api/orders`에 `totalPrice`를 직접 포함시켜, 클라이언트가 임의의 가격을 보내면 그대로 저장되는 보안 취약점을 발견 및 반려
4. 이 사건을 계기로 PR 리뷰 방법론을 정립: `git diff [base]...[head] --stat`로 먼저 변경 파일 목록과 규모를 확인하고, PR 설명과 실제 변경 파일이 일치하는지 대조한다. 특히 PR 설명에 없는 파일(`.env`, BE 라우터/모델 등)이 섞여 있으면 그 부분부터 우선적으로 확인한다
5. 같은 버그(`payment_status` 값 오타)가 서로 다른 브랜치(`feature/complete`, `feature/gift`)에서 각각 독립적으로 재발한 것을 발견, cherry-pick으로 필요한 부분만 선별 반영
6. 이미 실행 테스트까지 통과했던 M2 6단계(선물 사용 API) 코드가 이후 다른 브랜치 merge 과정에서 파일 전체가 옛 버전으로 덮어써지며 통째로 유실된 것을 발견
7. 새로 작성하는 대신 git 이력(`git log --oneline -- [파일경로]`)에서 유실 시점 직전 커밋을 찾아 코드를 복구하고, 최신 스키마와 호환되도록 재통합

### 결과

- M2 전체 기능(1~6단계)이 실행 테스트 기준으로 최종 정상 동작 확인
- PR 검토 원칙(diff 우선 확인, 설명과 실제 변경 파일 대조)을 체계화하여 이후 리뷰 속도와 정확도 개선
- 유실된 기능을 git 이력 기반으로 안전하게 복구하는 절차를 확립

## 문제 1. API와 ERD를 보완 방식으로 다시 설계하게 됨

### 상황

초기에 API와 ERD를 작성했지만 강사님 피드백 이후 대부분의 구조를 다시 검토하게 되었다.

### 원인

기획을 충분히 분석하지 않은 상태에서 API와 ERD를 먼저 작성했기 때문이었다.

### 해결 방법

- Figma를 다시 분석
- 필요한 화면만 선별
- 기능을 다시 분류
- 중복 기능 통합
- API 재설계
- API를 기반으로 ERD 재설계

### 느낀 점

기획을 충분히 분석한 이후 설계를 시작하면 이후 수정량이 크게 줄어든다는 것을 경험했다.

---

## 문제 2. FE와 BE의 응답 규격이 맞지 않아 화면이 출력되지 않음

### 상황

API는 정상적으로 응답했지만 프론트 화면에 상품이 표시되지 않았다.

### 원인
  - 필드명이 서로 달랐다.
  - 응답 구조가 명세서와 달랐다.
  - (근본 원인) 개발 시 API 명세서를 기준으로 구현하지 않았다.

### 해결 방법

- 필드명을 FE 기준으로 통일
- 응답 구조를 API 명세서 기준으로 수정

### 느낀 점
필드명과 같은 세부 요소는 기존 기획 단계에서 작성된 문서(API 명세서, ERD 등)를 기준으로 구현해야 한다는 점을 느꼈다. 공통된 기준 문서를 바탕으로 개발해야 서로 다른 필드명을 사용하는 등의 커뮤니케이션 오류를 예방하고, 불필요한 수정 작업도 줄일 수 있다.

## 문제 3. Products.js에서 Conflict 발생

### 상황

`Products.js` 파일에 대해 서로 다른 로직을 수정한 PR이 동시에 올라오면서 Merge Conflict가 발생하였다.

### 원인

- 동일한 코드 영역을 여러 작업자가 동시에 수정하면서 충돌이 발생하였다.
- BE에서 관리해야 하는 `Products.js` 파일을 FE에서도 함께 수정하여 역할 경계가 명확하지 않았다.
- 작업 브랜치가 최신 `develop` 내용을 충분히 반영하지 않은 상태에서 개발이 진행되었다.

### 해결 방법

- `develop` 브랜치의 최신 내용을 먼저 반영(`git pull`)한 후 Conflict를 해결하였다.
- API 응답 규격을 하나로 통일하여 코드를 정리하였다.
- 이후에는 **BE 영역은 BE, FE 영역은 FE가 담당하도록 역할을 명확히 구분**하기로 팀 내에서 합의하였다.
- 작업 시작 전 최신 브랜치를 반영하는 것을 협업 규칙으로 정리하였다.

### 느낀 점

Conflict 자체보다 **역할 분리와 최신 브랜치 동기화가 협업에서 훨씬 중요하다**는 것을 경험했다. 작업 범위를 명확하게 나누고, 개발을 시작하기 전에 `git pull`로 최신 상태를 유지하는 것만으로도 대부분의 충돌을 예방할 수 있을 것 같다고 느꼈다.

## 문제 4. 마일스톤 기획 문서와 API 명세서가 서로 다른 내용을 담고 있었음

### 상황

M2 마일스톤 문서에는 "결제 Mock", "금액 재계산"이라는 작업 항목이 명시되어 있었지만, 실제 API 명세서에는 이에 대응하는 요청/응답 필드가 존재하지 않았다.

### 원인

마일스톤 문서는 기획 초기에 작성된 방향성 중심의 문서였고, API 명세서는 이후 더 구체화되어 확정된 개발 계약서였다. 두 문서가 각자 다른 시점에 만들어지면서 동기화되지 않은 채 남아 있었던 것이 원인이었다.

### 해결 방법

- 불일치 내용을 정리하여 GitHub 이슈로 등록
- 강사님(기획 주체)에게 직접 확인을 요청하여, API 명세서를 기준으로 구현 범위를 확정
- 동시에 마일스톤 문서에는 없었지만 API 명세서에는 존재하던 유저 검색 API 누락도 함께 확인하여 작업 범위에 반영

### 느낀 점

이전 "문제 2"에서는 개발자가 명세서를 따르지 않아 문제가 발생했다면, 이번에는 **개발이 참고하는 문서들 자체가 서로 다른 말을 하고 있는 상황**이었다. "API 명세서를 기준으로 개발한다"는 원칙만으로는 부족했고, 명세서 역시 결국 기획에서 파생된 산출물이기 때문에 문서 간 충돌이 발견되면 기획 주체에게 직접 확인하는 절차가 개발 과정의 일부가 되어야 한다는 것을 배웠다.

## 문제 5. 최종 배포 DB에 seed 데이터가 중복/깨진 채로 삽입됨

### 상황

최종 배포 DB에 seed 데이터를 반영하는 과정에서 여러 번 실행을 시도하다 보니, 동일한 상품 데이터가 세 차례 중복 삽입되었고 그중 한 번은 한글이 깨진 채로 삽입되었다.

### 원인

- Windows 환경에서 cmd/PowerShell을 통한 SQL 파일 리다이렉션 과정에서 인코딩이 유실되어 일부 시도가 실패했다.
- seed.sql 파일 내에 세미콜론이 잘못 들어간 부분이 있어, 일부 INSERT 문이 조기 종료되며 SOURCE 실행이 중간에 실패하는 상황이 반복되었다.
- 여러 번의 재시도가 누적되며 중복/깨짐 데이터가 함께 쌓였다.

### 해결 방법

- mysql 클라이언트의 `SOURCE` 명령을 이용해 쉘 리다이렉션의 인코딩 문제를 우회
- seed.sql의 세미콜론 오타를 수정
- 기존에 쌓인 중복/깨진 데이터를 삭제한 뒤 수정된 seed.sql을 한 번만 재실행하여 정리

### 느낀 점

원인 파악 없이 재시도만 반복하면 문제가 해결되기는커녕 더 복잡하게 쌓인다는 것을 경험했다. 특히 Windows 환경에서는 쉘 리다이렉션 방식이 인코딩 문제를 일으키기 쉬우므로, DB에 파일을 반영할 때는 처음부터 클라이언트 내장 명령(`SOURCE`)을 사용하는 것이 더 안전하다는 것을 알게 되었다.

## 문제 6. `.env` 파일이 merge 과정에서 반복적으로 유실됨

### 상황

develop → main merge 과정에서 `.env` 파일이 삭제되는 커밋이 반영되어, 팀 전체와 개인 로컬 환경 모두에서 서버가 DB 접속 정보를 찾지 못해 500 에러가 반복적으로 발생하였다. 한 번 재생성한 이후에도, 다른 브랜치를 오가며 작업하는 과정에서 다시 사라지는 현상이 재발했다.

### 원인

- 프로젝트 초기 세팅 시 실수로 빈 `.env` 파일이 한 번 git에 커밋되었다가 나중에 삭제 처리된 이력이 있었다.
- `.env`는 `.gitignore`에 등록되어 있어 git이 추적하지 않지만, 한 번이라도 추적 이력이 있었던 파일이 삭제 커밋으로 반영되면 로컬의 실제 파일도 함께 사라지는 동작이 발생했다.
- `.gitignore` 설정 자체에는 문제가 없었으나, 이미 벌어진 삭제 이력을 인지하지 못한 채 작업을 이어가며 동일한 증상이 반복되었다.

### 해결 방법

- 원인을 git 이력(`git log --all -- .env`, `git show [커밋]:.env` 등)으로 추적하여 민감정보 노출 여부를 먼저 확인 (노출 없음을 확인)
- 로컬 및 EC2 배포 서버에 `.env`를 동일한 접속 정보로 재생성
- 팀 전체에 재발 가능성을 공지하고, 각자 로컬 `.env` 존재 여부를 확인하도록 안내

### 느낀 점

`.env`처럼 git으로 보호되지 않는 파일은 브랜치를 자주 오가는 작업 중에 예상치 못하게 사라질 수 있다는 것을 경험했다. 문제가 재발했을 때 "또 안 되네"라고 넘기기보다, 매번 `git log`로 근본 원인을 재확인하는 습관이 필요하다는 것을 느꼈다.

## 문제 7. 세션 인증에 필요한 `credentials` 옵션이 여러 PR에서 반복 누락됨

### 상황

세션(쿠키) 기반 인증을 사용하는 프로젝트임에도, 로그인/회원가입 화면과 주문 페이지를 담당한 두 명의 서로 다른 작업자(FE, Cloud)가 각각 독립적으로 `fetch` 호출에 `credentials: 'include'` 옵션을 빠뜨리는 동일한 실수를 반복하였다.

### 원인

- 이 옵션이 없으면 세션 쿠키가 전송되지 않아, 로그인된 상태에서도 인증이 필요한 API가 401을 반환하게 되는 fetch API 자체의 스펙 디테일이었다.
- API 명세서에는 "세션 기반 인증이므로 credentials가 필요하다"는 내용이 별도로 문서화되어 있지 않았다.

### 해결 방법

- PR 리뷰 시 fetch 호출부를 diff로 직접 확인하여 누락을 발견, 수정 요청
- 발견 즉시 개별 코멘트로 대응하되, 같은 문제가 반복된 시점부터는 팀 전체에 공통 안내로 전달

### 느낀 점

같은 실수가 서로 다른 사람에게서 반복된다는 것은 개인의 부주의가 아니라, 프로젝트 차원에서 이 부분에 대한 가이드 문서가 없었다는 신호로 봐야 한다는 것을 깨달았다. API 명세서가 다루지 않는 "구현 관점의 주의사항"은 별도의 개발 가이드 문서로 남겨두어야 한다는 필요성을 느꼈다.

## 문제 8. FE가 BE 코드(라우터/모델)를 직접 수정하며 발생한 보안 취약점

### 상황

FE가 주문 페이지에 수량 선택 기능을 구현하는 과정에서, `routes/orders.js`와 `db/models/orderModel.js`(BE 파일)를 직접 수정하여 PR에 포함시켰다. 이 과정에서 클라이언트가 보낸 `totalPrice` 값을 서버가 검증 없이 그대로 저장하도록 바뀌어 있었다.

### 원인

- FE가 화면에 필요한 데이터를 빠르게 구현하기 위해, "문제 3"에서 이미 합의했던 BE-FE 역할 분리 원칙을 벗어나 BE 파일까지 직접 수정하였다.
- 서버가 클라이언트가 보낸 금액을 그대로 신뢰하는 구조로 바뀌면서, 클라이언트가 임의의 가격(예: 1원)을 보내 주문 금액을 조작할 수 있는 보안 취약점이 발생했다.
- "수량(quantity)" 필드 자체도 API 명세서와 ERD에 존재하지 않는 필드였다.

### 해결 방법

- 즉시 PR을 반려하고, 서버가 `productId`로 상품 가격을 직접 조회하는 원래 구조로 되돌림
- 이 사건을 계기로 PR 리뷰 시 `git diff [base]...[head] --stat`로 먼저 변경 파일 목록과 규모를 확인하는 절차를 정립. PR 설명(예: "UI만 수정했다")과 실제 변경 파일이 일치하는지 대조하고, 설명에 없는 파일(`.env`, BE 라우터/모델 등)이 섞여 있으면 그 부분부터 우선적으로 확인하는 방식으로 리뷰 방법론을 구체화함
- 이후 BE 파일이 FE PR에 포함된 경우, 해당 부분만 BE가 직접 가져가서 재작업하는 방식으로 대응

### 느낀 점

기능 구현 자체는 문제없어 보여도, 클라이언트가 보낸 값을 서버가 그대로 신뢰하는 구조는 보안 취약점으로 직결될 수 있다는 것을 체감했다. 또한 PR 설명만 믿지 않고 실제 diff를 항상 먼저 확인하는 습관이 왜 중요한지 이 사건을 통해 명확히 이해하게 되었다.

## 문제 9. 같은 버그(`payment_status` 오타)가 서로 다른 브랜치에서 독립적으로 재발

### 상황

한 작업자가 선물함/바코드 사용 화면을 두 개의 서로 다른 브랜치(`feature/complete`, `feature/gift`)에서 각각 병행 작업하였고, 한쪽 브랜치에서 이미 수정했던 `payment_status` 값 오타(`'COMPLETED'` → `'paid'`)가 다른 브랜치에는 그대로 남아있는 상태로 재등장하였다.

### 원인

- 같은 기능을 여러 브랜치에서 독립적으로 작업하면서 최신 수정 사항이 서로 동기화되지 않았다.
- develop을 병합하지 않은 오래된 브랜치 기준으로 작업이 이어졌다.

### 해결 방법

- 두 브랜치의 diff를 대조하여 겹치는 부분(버그가 남아있는 부분)과 새로운 부분(순수 UI 개선)을 구분
- `git checkout [브랜치] -- [파일]` 방식으로 필요한 파일만 선별적으로 cherry-pick하여 develop에 반영
- cherry-pick 이후에도 가져온 파일 안에 또 다른 회귀(API 메서드, 상태값 등)가 없는지 재검증

### 느낀 점

여러 브랜치가 유사한 기능을 병행 작업할 때는, 병합 전 반드시 대상 파일 전체를 diff로 대조해서 "새로운 개선"과 "이미 해결한 문제의 재등장"을 구분해야 한다는 것을 배웠다. 겉보기에 같은 파일이라도 어느 시점에서 분기됐는지에 따라 내용이 크게 다를 수 있다.

## 문제 10. 이미 완성했던 기능(6단계)이 merge 과정에서 통째로 유실됨

### 상황

M2 6단계에서 이미 구현하고 실행 테스트로 200 성공 응답까지 확인했던 `GET /api/gifts/:id`, `PATCH /api/gifts/:id/use` 라우터가, 이후 다른 PR이 병합되는 과정에서 `routes/gifts.js` 파일 전체가 6단계 이전 버전으로 덮어써지며 통째로 사라졌다.

### 원인

- FE가 화면에 필요한 필드(`brand`, `usedAt`)를 추가하기 위해 `routes/gifts.js`를 수정한 PR을 올렸는데, 이 PR의 기반 브랜치가 6단계 병합 이전 시점에서 분기된 오래된 상태였다.
- 이 PR이 develop에 병합되면서, 이미 완성되어 있던 6단계 라우터 코드가 통째로 그 이전 버전으로 되돌아갔다.

### 해결 방법

- 처음에는 "라우터가 아예 구현된 적이 없다"고 잘못 판단할 뻔했으나, 이전 실행 테스트 기록과 모순됨을 인지하고 재조사
- `git log --oneline -- routes/gifts.js`로 파일의 전체 커밋 이력을 확인하여, 라우터가 존재했던 마지막 커밋을 특정
- 해당 커밋의 코드를 그대로 복사하지 않고, 최신 스키마(`brand`, `usedAt` 필드, `payment_status` 조건 등)와 호환되는지 검토한 뒤 재통합
- develop을 직접 수정하지 않고 별도 브랜치(`fix/BE-api`)를 만들어 작업 후 병합

### 느낀 점

"기능이 없다"는 결론을 내리기 전에, 정말 처음부터 없었던 것인지 아니면 있었는데 사라진 것인지를 git 이력으로 먼저 확인해야 한다는 것을 배웠다. 이미 검증된 코드를 다시 찾아 복구하는 것이, 새로 처음부터 작성하는 것보다 훨씬 안전하고 빠른 경우가 많다는 것도 체감했다.

# 작업하면서 알게 된 점

### M0~M0-1

이번 프로젝트를 진행하면서 가장 크게 느낀 점은 **개발보다 기획의 중요성이 훨씬 크다**는 것이었다.

처음에는 API 명세서와 ERD를 단순히 개발 전에 작성하는 문서 정도로 생각했다. 하지만 실제 프로젝트를 진행하면서 기획 단계가 잘 되어 있을수록 개발 과정에서 발생하는 수정과 소통 비용이 크게 줄어든다는 것을 직접 경험했다.

이번 프로젝트에서는 다음과 같은 순서로 기획을 진행하였다.

> **Figma(UI + 기능 정의) 분석 → API 명세서 작성 → ERD 설계**

제공된 Figma 기획서를 통해 필요한 화면과 기능을 먼저 정의하고, 이를 기반으로 API를 설계한 뒤 API를 기준으로 ERD를 작성하니 개발 과정에서 구현해야 할 내용이 명확해졌다.

반대로 기획이 부족하면 FE, BE, DB가 서로 다른 기준으로 개발하게 되어 반복적인 수정과 소통이 계속 발생한다.

### M1~M2

M1 구간에서는 **좋은 개발은 좋은 기획에서 시작된다**는 것을 다시 한번 확인했다면, M2를 진행하며 한 가지를 더 배웠다. **기획 문서 자체도 항상 완벽하거나 일관되지는 않는다는 것**이다.

마일스톤 문서와 API 명세서가 서로 다른 내용을 담고 있던 상황에서, 어느 한쪽을 임의로 우선시하기보다 기획 주체에게 직접 확인하는 절차를 거친 것이 결과적으로 옳은 선택이었다. 개발자의 역할은 문서를 그대로 따르는 것에 그치지 않고, 문서 간 충돌이나 모호함을 발견했을 때 이를 명확히 하는 과정까지 포함한다는 것을 이번 단계에서 체감했다.

### M2 후반

M2 후반부에는 새로운 기능을 만드는 것보다 **이미 만든 것을 지키는 일**이 더 큰 비중을 차지했다. 같은 버그가 서로 다른 사람, 서로 다른 브랜치에서 반복되는 것을 보면서, 문제가 한 번 해결됐다고 끝난 게 아니라 팀 전체에 공유되고 문서화되지 않으면 언제든 재발할 수 있다는 것을 알게 되었다.

또한 PR 설명만 믿고 넘어가지 않고, `git diff`로 실제 변경 내용을 항상 먼저 확인하는 습관이 이 시기부터 가장 중요한 작업 방식으로 자리 잡았다. 특히 FE가 BE 영역을 직접 수정하는 경우가 반복되면서, "설명과 실제 변경 파일이 일치하는지"를 확인하는 절차 자체가 하나의 검증 루틴이 되었다.

# 작업 Tip

- 💡 AI는 코드 생성뿐만 아니라 공지 작성, 문서화, 코드 리뷰, 로직 설계 등 프로젝트 전반에서 적극적으로 활용하는 것이 생산성을 크게 높여준다.

- 💡 Antigravity와 같은 전문 AI에게 질문하기 전에 ChatGPT와 같은 범용 AI를 활용하여 질문 자체를 다듬으면 원하는 답변을 받을 확률이 훨씬 높아진다.

- 💡 API 명세서와 ERD를 충분히 검토한 후 개발을 시작하면 불필요한 수정 작업을 크게 줄일 수 있다.

- 💡 구현보다 문서를 먼저 확인하는 습관이 협업 과정에서 큰 도움이 된다.

- 💡 AI가 "완료했다", "검증했다"고 요약해서 보고하더라도, 실제 실행 로그나 코드 원문을 직접 확인하기 전까지는 완전히 신뢰하지 않는 것이 안전하다. 요약과 실제 결과가 다른 경우가 종종 있었다.

- 💡 Windows 환경에서 한글이 포함된 SQL 파일을 DB에 반영할 때는 `mysql ... < file.sql` 같은 리다이렉션 방식보다, mysql 클라이언트 접속 후 `SOURCE` 명령을 사용하는 것이 인코딩 문제를 피하는 데 더 안정적이다.

- 💡 기획 문서와 API 명세서처럼 서로 다른 시점에 작성된 문서 간에는 불일치가 생길 수 있으므로, 개발 착수 전에 두 문서를 한 번씩 대조해보는 습관이 이후 재작업을 줄여준다.

- 💡 PR을 리뷰할 때는 `git diff [base]...[head] --stat`로 먼저 변경 파일 목록과 규모를 확인하고, PR 설명과 실제 변경 파일이 일치하는지 대조한다. 예를 들어 "UI만 수정했다"는 설명인데 `.env`나 BE 라우터/모델 파일이 변경 목록에 포함되어 있다면, 그 부분부터 우선적으로 확인해야 한다.

- 💡 같은 버그가 서로 다른 PR이나 브랜치에서 반복 발견된다면, 개인의 실수로만 보지 말고 팀 차원의 가이드 문서 공백이나 브랜치 동기화 문제를 함께 의심해봐야 한다.

- 💡 "기능이 없다"고 결론 내리기 전에, `git log --oneline -- [파일경로]`로 정말 처음부터 없었던 것인지 아니면 있었는데 유실된 것인지 먼저 확인하는 습관이 불필요한 재작업을 막아준다.

# 주의사항

- ⚠️ `.env` 파일은 절대 Git에 커밋하지 않는다.

- ⚠️ Git 브랜치 전략을 명확하게 이해한다.

- ⚠️ 공통된 작업에 대해 테스크 분리를 명확하게 한다.

- ⚠️ 문서 간 불일치를 발견했을 때는 개발자가 임의로 해석하지 말고, 기획 주체에게 직접 확인 후 진행한다.

- ⚠️ DB에 데이터를 반영하는 작업(특히 대량 INSERT, 초기화)은 실행 전 팀에 공지하고, 결과를 반드시 재조회로 검증한다.

- ⚠️ FE가 BE 파일(라우터, 모델)을 직접 수정한 PR은 반드시 별도로 확인하고, 필요시 BE가 직접 가져가서 재작업한다. 클라이언트가 보낸 값(가격, 수량 등)을 서버가 검증 없이 그대로 저장하지 않도록 항상 주의한다.
