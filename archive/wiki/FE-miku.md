# 작업 중 발생한 이슈

## 1. API 설계 및 명세서 작성

### 상황

* **프로젝트에서 사용할 API 명세서를 작성하기 위해 화면별 기능을 분석**

  * M2까지 필요한 화면을 선별
  * 화면별 기능을 정리하고 API 적용 여부를 검토
  * 공통 기능을 분석하여 중복 여부를 확인
  * FE·BE가 동일한 기준으로 개발할 수 있도록 API 명세서를 작성

### 원인

* **API 설계 경험 부족으로 기능 분리와 데이터 설계 기준이 명확하지 않았음**

  * API와 프론트엔드 기능의 역할 구분이 어려웠음
  * Request/Response 데이터 구조를 설계하는 경험이 부족
  * 필드명과 데이터 구조에 대한 프로젝트 규칙이 정립되지 않았음

### 해결 방법

* **기능 중심으로 API를 재설계하고 리뷰를 통해 구조를 개선**

  * API 설계 예시를 참고하여 REST API 구성 방식을 학습
  * 화면이 아닌 기능 단위로 API를 설계
  * 공통 기능은 하나의 API로 통합하여 중복을 최소화
  * 코드 리뷰와 피드백을 반영하여 필드명과 데이터 구조를 일관성 있게 수정

### 작업하면서 느낀 점

* **API는 개발 전에 충분히 설계할수록 협업 효율이 높아진다는 것을 배움**

  * 화면보다 기능을 기준으로 API를 설계하는 것이 유지보수에 유리
  * 명세서는 개발 과정에서 지속적으로 검토하고 수정해야 완성도를 높일 수 있었음
  * 필드명과 데이터 구조를 미리 통일하면 FE와 BE 간의 연동 오류를 줄일 수 있었음

## 2. FE와 BE의 응답 규격 불일치로 화면이 출력되지 않음

### 상황

* **홈 화면에 상품 목록을 API와 연동하는 과정에서 문제가 발생**

  * FE에서 API를 호출하여 데이터를 받아옴
  * Network 탭에서는 정상 응답을 확인
  * 응답 데이터가 화면에 바인딩되지 않아 상품이 표시되지 않았음

### 원인

* **API 응답 형식이 명세서 및 FE 구현과 일치하지 않았음**

  * FE와 BE의 필드명이 서로 달랐다.
  * 응답 구조(Response)가 API 명세서와 다르게 구현되어 있었음
  * 개발 과정에서 API 명세서를 기준으로 구현되지 않음

### 해결 방법

* **API 명세서를 기준으로 응답 규격을 통일**

  * FE와 BE의 필드명을 동일하게 수정
  * API 응답 구조를 명세서에 맞게 변경
  * 수정 후 화면에서 상품 데이터가 정상적으로 출력되는 것을 확인

### 작업하면서 느낀 점

* **API 명세서를 기준으로 FE와 BE가 동일한 규격을 유지하는 것이 중요하다는 것을 배움**

  * API가 정상 응답하더라도 응답 규격이 다르면 화면에 데이터가 표시되지 않을 수 있다는 점을 확인
  * 개발 초기부터 필드명과 응답 구조를 통일하면 불필요한 디버깅 시간을 줄일 수 있었음
  * API 변경 사항은 FE와 BE가 함께 공유하고 명세서를 최신 상태로 유지하는 것이 중요하다는 점을 느낌

## 3. `Products.js` 파일에서 Merge Conflict 발생

### 상황

* **상품 기능을 개발하고 PR을 병합하는 과정에서 충돌이 발생**

  * FE와 BE가 모두 `Products.js` 파일을 수정
  * 작업 브랜치가 최신 `develop` 브랜치를 충분히 반영하지 않은 상태에서 개발이 진행
  * PR 병합 과정에서 Merge Conflict가 발생

### 원인

* **역할 구분과 브랜치 관리가 명확하지 않아 충돌이 발생**

  * 동일한 코드 영역을 여러 작업자가 동시에 수정
  * BE에서 관리해야 하는 `Products.js` 파일을 FE에서도 수정하여 역할 경계가 명확하지 않았음
  * 최신 `develop` 브랜치를 반영하지 않은 상태에서 작업을 진행

### 해결 방법

* **최신 브랜치를 반영하고 역할을 명확히 구분하여 문제를 해결.**

  * `develop` 브랜치의 최신 내용을 반영한 후 Merge Conflict를 해결
  * API 응답 규격을 하나로 통일하여 코드를 정리
  * FE와 BE의 담당 영역을 명확히 구분하기로 팀 내에서 합의
  * 작업 시작 전 최신 `develop` 브랜치를 반영하는 것을 협업 규칙으로 정함

### 작업하면서 느낀 점

* **Merge Conflict는 역할 분리와 브랜치 관리만으로도 대부분 예방할 수 있다는 것을 배웠음**

  * 작업 범위를 명확히 구분하면 동일한 파일 수정으로 인한 충돌을 줄일 수 있었음
  * 개발을 시작하기 전에 최신 `develop` 브랜치를 반영하는 습관이 중요하다는 것을 느낌
  * 협업 규칙을 미리 정하고 공유하는 것이 개발 효율과 안정성을 높인다는 점을 경험
